# Response 工具后最终回复流式修复（69）

## 问题

模型在调用工具时，过程文本可以正常流式显示；工具完成后的最终回答虽然已经按分块到达主进程，却一直保存在 `currentStepText`。只有 AI SDK 的 `onStepEnd` 确认该步骤没有新的工具调用后，运行时才把整段文本写入 `finalText`。

因此界面表现为：

1. 工具活动正常更新；
2. 最终回答生成期间没有在回答区域逐字显示；
3. 步骤结束时整段回答一次出现。

## 修复

新增临时步骤角色：

- `commentary`：普通工具前说明；
- `final_candidate`：已有工具执行记录后产生的下一段模型文本；
- `none`：没有活动文本。

工具后的下一模型步骤会先作为 `final_candidate` 实时显示在最终回答区域。如果该步骤随后又发起工具调用，步骤结束时仍会按原规则归入过程说明；如果没有工具调用，则无缝转为正式 `finalText`。

该角色只影响实时展示，不会提前把候选文本写成最终消息，也不会改变 Tool Runtime、Checkpoint 或对话持久化语义。

## 涉及界面

- Response：工具后的回答候选直接在最终回答区域流式显示；
- Conversation：实时 Assistant 消息同步显示相同的回答候选，不再只放在 Thinking Timeline 中。

## 回归覆盖

- 无工具的普通回复保持原有流式行为；
- 工具前说明仍显示在活动区域；
- 工具后的最终回答候选逐块显示；
- 若候选步骤再次调用工具，可回归为 commentary；
- 独立 finalization 阶段继续逐块更新 `finalText`。
